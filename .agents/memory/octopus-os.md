## Architecture

- **Frontend**: `artifacts/mockup-sandbox` (React + Vite + Tailwind) — serves the full OS dashboard at `/__mockup/`
- **Backend**: `artifacts/api-server` (Express + TypeScript) — serves at `/api/`
- **DB**: PostgreSQL via Drizzle ORM in `lib/db`

## Auth

- JWT-based (jsonwebtoken + bcryptjs), 7-day token expiry
- Token stored in localStorage under `octopus_token` / `octopus_user`
- `JWT_SECRET` defaults to hardcoded string — must set env var in production
- `requireAuth` middleware in `artifacts/api-server/src/middleware/auth.ts`

## DB Tables

- `users` — id, email, password (bcrypt), name, role
- `ai_providers` — per-user AI provider configs (openai/gemini/claude/etc)
- `social_accounts` — per-user social platform connections
- `affiliate_networks` — per-user affiliate network configs
- `campaigns` — per-user campaign tracking
- `events` — durable store for `@workspace/event-bus`; every published event, persisted before dispatch (audit trail + replay source)
- `tasks` — durable store for `@workspace/task-queue`; doubles as Task Manager lifecycle record and Queue Manager claim/lock state (see OS Core Modules below)

## OS Core Modules

- **Event Bus** (`lib/event-bus`, `@workspace/event-bus`) — the only inter-module communication channel. `EventBus.publish(type, source, payload, options)` persists then fans out to subscribers, isolating handler failures per-subscriber. Wired as a singleton in `artifacts/api-server/src/lib/event-bus.ts`, exposed read-only via `GET /api/system/events`.
- **Task Manager + Queue Manager** (`lib/task-queue`, `@workspace/task-queue`) — durable async work handoff, built on the Event Bus. `TaskQueue.enqueue()` creates a task and publishes `task.created`; a worker `claim()`s it (atomic via Postgres `SELECT ... FOR UPDATE SKIP LOCKED`, safe across processes), then `complete()`s or `fail()`s it — failures retry with exponential backoff up to `maxAttempts`, then move to `failed`. Every transition (`task.started`/`completed`/`failed`/`retrying`/`cancelled`/`reclaimed`) is a published event. Platform-independent: depends only on a `TaskStore` interface (`DrizzleTaskStore` in production, `InMemoryTaskStore` for tests) and a local `EventPublisher` interface, not on Express or `@workspace/event-bus` directly. Wired as a singleton in `artifacts/api-server/src/lib/task-queue.ts`, exposed via `POST/GET /api/tasks`, `GET/POST /api/tasks/:id`, `/api/tasks/:id/cancel`, `/api/tasks/claim`, `/api/tasks/:id/complete`, `/api/tasks/:id/fail`. A periodic `reclaimStale()` sweep in `index.ts` requeues tasks abandoned by crashed workers.
- **Brain** (`lib/brain`, `@workspace/brain`) — OS Core's decision maker. `Brain.registerRule(rule)` registers a `DecisionRule` (name + event pattern + optional `priority` + `evaluate()`); a single Event Bus subscription (`"*"`, handler name `"os-core:brain-dispatch"`) feeds every incoming event through local pattern matching against all registered rules, so every rule interested in an event is evaluated together, not in isolation. Every matching rule durably records a `BrainDecision` via `BrainStore` (`DrizzleBrainStore` in production, `InMemoryBrainStore` for tests): `noop` if the rule declined, `action_failed` if it threw or its action failed, `superseded` if the `DecisionEngine` picked a different rule's candidate for the same event, `actioned` once the winner's action succeeds. Actions go through local `TaskEnqueuer`/`EventPublisher` interfaces, satisfied structurally by `TaskQueue`/`EventBus` with no compile-time dependency on either package. The Brain never re-evaluates rules against events it authored itself (source `"brain:*"`/`"os-core:brain"`), which prevents `publish_event` rules and its own `brain.decision.made` broadcast from looping. Ships with one core rule today: `task-failure-alert` (`artifacts/api-server/src/lib/brain-rules.ts`), which publishes `brain.alert.task_failed` whenever a task exhausts its retries. Wired as a singleton in `artifacts/api-server/src/lib/brain.ts` (constructed with a `DecisionEngine`), rules registered at startup in `index.ts`, exposed read-only via `GET /api/brain/decisions` and `GET /api/brain/decisions/:id`.
- **Decision Engine** (`lib/decision-engine`, `@workspace/decision-engine`) — pure arbitration library, zero dependency on events/tasks/the Event Bus. `DecisionEngine.arbitrate(candidates: ArbitrationCandidate[])` delegates to a swappable `ArbitrationStrategy` (`setStrategy()` for runtime reconfiguration) and defensively drops anything a misbehaving strategy fabricates that wasn't in its input. Ships `allMatchStrategy` (every candidate acts — the Brain's default), `firstMatchStrategy` (first-registered wins), and `priorityStrategy` (highest `priority` wins, ties go to first-registered). `Brain` is the only consumer so far, constructed with `new DecisionEngine(allMatchStrategy)` in `artifacts/api-server/src/lib/brain.ts`.
- **Rule Engine** (`lib/rule-engine`, `@workspace/rule-engine`) — data-defined rules on top of the Brain, decoupled from `@workspace/brain` at compile time the same way `Brain` is decoupled from `@workspace/event-bus` (`RuleEvent`/`RuleOutcome` mirror `BrainEvent`/`RuleDecision` structurally; `RuleRegistrar` is the minimal `{ registerRule }` capability `Brain` satisfies). A `RuleDefinition` (name, event pattern, JSON `Condition` tree, `ActionTemplate`, priority, enabled) is stored in the `rule_definitions` table and compiled via the pure function `compileRule()` into a `CompiledRule`; `RuleEngine.create/update/enable/disable/delete` keep a running `Brain`'s registered rules in sync automatically (register/unregister, never in-place mutation, since `Brain` doesn't support that). `Condition` supports `eq`/`neq`/`gt`/`gte`/`lt`/`lte`/`contains`/`exists` leaf nodes composed with `and`/`or`/`not`, evaluated via dot-path field resolution (`resolveField`) against the triggering event. `ActionTemplate` payloads support whole-token templating only (`renderTemplate`: a string that is *exactly* `"{{dot.path}}"` is replaced with that resolved value; everything else is left as a literal) — intentionally not a general expression language. Wired as a singleton in `artifacts/api-server/src/lib/rule-engine.ts` (constructed with the `brain` singleton as its `RuleRegistrar`), `loadAndSync()` called at startup in `index.ts` after `registerCoreRules(brain)`, exposed via full CRUD at `/api/rules*` (`routes/rules.ts`) — mutating endpoints admin-only, reads open to any authenticated user.
- **Agent Manager** (`lib/agent-manager`, `@workspace/agent-manager`) — agent registry, lifecycle, and durable run history, with no knowledge of how to actually run an agent. `AgentDefinition` (name, instructions, capabilities: string[], status active/disabled) lives in the `agents` table; every invocation produces an `AgentRun` (`agent_runs` table) recording status running→completed/failed, input, output, error, timestamps — recorded once execution starts, regardless of outcome. `AgentManager.invoke()` throws `AgentNotInvocableError` for a missing/disabled agent (no run recorded, nothing was attempted) or `AgentExecutorNotConfiguredError` if no `AgentExecutor` is wired up (also no run recorded). As of the AI Provider Manager module, a real executor IS wired up in production (see below) — `AgentExecutorNotConfiguredError`/`501` is now only reachable if that wiring is ever removed. Full CRUD + enable/disable at `/api/agents*`, mutating endpoints admin-only, invoke and reads open to any authenticated user.
- **AI Provider Manager** (`lib/ai-provider-manager`, `@workspace/ai-provider-manager`) — provider configuration (`provider_configs` table — distinct from the pre-existing, untouched `providersTable`/`ai_providers` table that backs the legacy per-user `/api/providers*` routes) plus a uniform `ProviderClient.complete()` seam. Ships a real `AnthropicProviderClient` calling Anthropic's Messages API (`buildAnthropicRequest`/`parseAnthropicResponse` are pure and unit-tested directly; the HTTP call uses an injectable `fetch`, defaulting to the runtime global, purely so tests can exercise request-building without a live network call). `AiProviderManager` registers `ProviderClientFactory`s per `providerType` (`anthropic` built in, others via `registerFactory()`), resolves a config to a client, and exposes `complete(configId, request)`/`completeWithDefault(request)` (throws `NoDefaultProviderError` if no config has `isDefault: true`). `ProviderBackedAgentExecutor` is the adapter that makes this module's manager satisfy `@workspace/agent-manager`'s `AgentExecutor` structurally (mirrors `AgentDefinition`/`AgentExecutor` as `AgentLike`/`AgentLikeExecutor`, same decoupling pattern as Rule Engine ↔ Brain) — it turns `agent.instructions` into the system prompt and the invocation `input` into a user message (string input used verbatim; an object with a `message`/`prompt`/`input` string field extracted the same way; anything else JSON-stringified). Wired into `AgentManager`'s constructor in `artifacts/api-server/src/lib/agent-manager.ts` — the *only* execution path an agent invocation takes, no parallel path. Admin-only CRUD at `/api/provider-configs*`.
- **Tool Manager** (`lib/tool-manager`, `@workspace/tool-manager`) — tool registration, schema-validated invocation, and an in-process handler registry, following the same registration/execution split as Agent Manager/AI Provider Manager. `ToolDefinition` (name — unique, the invocation key — description, `inputSchema`, `handlerName`, status) lives in the `tools` table. `inputSchema` is a deliberately minimal JSON-Schema-like structure (`JsonSchema`: type/enum/properties+required for objects/items for arrays — no `$ref`/`oneOf`/custom formats), validated via the pure `validateAgainstSchema()` on every `invoke()` before a handler ever runs, collecting every violation rather than stopping at the first. `ToolManager.registerHandler(name, handler)` registers the actual code a tool runs — never reconstructed from stored data; `handlerName` is not checked to have a registered handler at create/update time, only at `invoke()` (same precedent as `AiProviderConfig.providerType`). `invoke()` throws distinct errors for every pre-execution failure (`ToolNotFoundError`, `ToolDisabledError`, `ToolInputValidationError`, `UnknownToolHandlerError`) — none of those produce a `tool.invoked`/`tool.failed` broadcast; once a handler runs, success or failure always best-effort broadcasts through a decoupled `EventPublisher` (mirrors `EventBus.publish` exactly, same pattern as Brain/TaskQueue). Wired as a singleton on the same `eventBus` instance in `artifacts/api-server/src/lib/tool-manager.ts`, with zero handlers registered by default — concrete tools get their handlers registered by whatever integrates them. Admin-only CRUD/lifecycle + open invoke at `/api/tools*`. `routes/agents.ts` validates an agent's `capabilities` array against `toolManager.list({status:"active"})` on create/update, reusing the Tool Manager's own registry rather than duplicating a lookup.
- **Workflow Engine** (`lib/workflow-engine`, `@workspace/workflow-engine`) — ordered, no-branching multi-step orchestration (`workflow_definitions`/`workflow_runs` tables). A `WorkflowStep` is a discriminated union (`tool`/`agent`/`task`/`event`), each delegating to the real module that owns that capability through its own decoupled interface: `ToolInvoker` ≈ `ToolManager.invoke`, `AgentInvoker` ≈ `AgentManager.invoke` (checks `result.status === "failed"` and throws, since `AgentRun.status` can be `failed` without the call itself throwing), `TaskEnqueuer` ≈ `TaskQueue.enqueue` (task steps are fire-and-forget — the run doesn't wait for `task.completed`), `EventPublisher` ≈ `EventBus.publish`. `WorkflowEngine.run()` builds a context `{ input, steps: { <name>: { output } } }` and threads it through a step-local, whole-token-only `renderTemplate()` (own implementation — not shared with Rule Engine's, since the context shapes are unrelated) so later steps can reference `"{{steps.<name>.output...}}"`/`"{{input...}}"`. Stops at the first failing step; every step before it keeps its recorded `StepResult`, and the `WorkflowRun` is marked `failed` naming which step and why. A step type with no configured dependency throws `WorkflowDependencyMissingError` rather than skipping silently. Wired as a singleton spanning `taskQueue`+`eventBus`+`agentManager`+`toolManager` simultaneously in `artifacts/api-server/src/lib/workflow-engine.ts` — the first OS Core module to depend on four others at once, still with zero direct compile-time imports between them. Admin-only CRUD/lifecycle + open run/run-history at `/api/workflows*`.
- **Scheduler** (`lib/scheduler`, `@workspace/scheduler`) — cron/interval-triggered workflow runs or task enqueues (`scheduled_jobs`/`scheduled_job_runs` tables). Ships a real, from-scratch 5-field cron parser (`parseCronExpression`: `*`, single values, comma lists, ranges, step suffixes on a wildcard or range) and `getNextRunTime` (brute-force minute-stepping forward from a given instant, UTC, up to ~4 years — the standard approach since cron is minute-granular). `computeNextRunAt(schedule, from)` handles both `{type:"interval", intervalMs}` and `{type:"cron", expression}`. `ScheduledJob.nextRunAt` is precomputed on create/update (recomputed only if `schedule` changes) so `tick()`'s "what's due" check is a plain `nextRunAt <= now` filter, not a live cron evaluation. `tick()` dispatches each due job through `WorkflowRunner`≈`WorkflowEngine.run` or `TaskEnqueuer`≈`TaskQueue.enqueue` (decoupled, same pattern as Workflow Engine's own interop), records a `ScheduledJobRun` whether it succeeds or fails (a workflow run reporting `status:"failed"` counts as a job failure even though the call itself didn't throw), and always reschedules `nextRunAt` for the next occurrence regardless of outcome. `start(intervalMs=30000)`/`stop()` wrap `tick()` in an `unref()`'d `setInterval`; the constructor's `now` parameter is injectable so tests call `tick()` directly without real timers. No distributed lock — single-process assumption, documented. Wired in `artifacts/api-server/src/lib/scheduler.ts` on top of `workflowEngine`+`taskQueue`, started via `scheduler.start()` in `index.ts` after every other module and `scheduler.stop()` on graceful shutdown. Admin-only CRUD/lifecycle/run-history at `/api/scheduled-jobs*`.
- **Notification System** (`lib/notification-system`, `@workspace/notification-system`) — channel configuration (`notification_channels` table) and delivery tracking (`notification_deliveries` table — doubles as the in-app inbox: `recipientUserId`+message+`read` is everything `GET /api/notifications` needs, so there is no separate notifications table). Ships two real channels: `in_app` (a no-op `send()` — the delivery record itself is the whole point) and `webhook` (a genuine HTTP POST of title/body/data as JSON, injectable `fetch` for testing request construction without a live call). `NotificationManager.send()` resolves `channelType` through a factory registry (`registerChannel()` for anything beyond the two built-ins — same registry pattern as `AiProviderManager`), always records a `NotificationDelivery` (`sent`/`failed`) once a channel is actually invoked, and throws `NotificationChannelNotFoundError`/`NotificationChannelDisabledError`/`UnknownNotificationChannelTypeError` before that point (no delivery recorded for any of those, since nothing was attempted). Optionally subscribes (constructor-injected, decoupled `EventSubscriber` ≈ `EventBus.subscribe`) to a `notification.requested` event type so any module can trigger a send by publishing an event with a channelConfigId/message/recipientUserId payload, with zero compile-time dependency on this package; a malformed payload is logged and ignored, never thrown. Wired in `artifacts/api-server/src/lib/notification-manager.ts` on the same `eventBus` instance, imported for its constructor side-effect (the subscription) in `index.ts`. Admin-only channel CRUD at `/api/notification-channels*`; `/api/notifications/send`, the inbox listing, and mark-read are open to any authenticated user, scoped to their own `recipientUserId` unless the caller is an admin.
- **Audit & Observability** (`lib/audit-observability`, `@workspace/audit-observability`) — two separate pieces. `AuditLogger` writes to its own `audit_log` table; it is never called directly by any route file — `artifacts/api-server/src/middleware/audit.ts` is a global Express middleware (applied in `app.ts` before the router) that records every successful (2xx/3xx) POST/PUT/PATCH/DELETE automatically, deriving `action`/`resourceType` from the *matched route pattern* (`req.route.path`, never the raw URL, so a UUID never becomes part of an action name) and the actor from `req.user` (read inside a `res.on("finish")` listener, so it reflects whatever `requireAuth` populated earlier in the same request's chain). `ObservabilityService` has no knowledge of any other module — `registerProbe()` takes a `{name, collect()}` pair, and `artifacts/api-server/src/lib/audit-observability.ts` registers one per module (Event Bus → `eventBus.history()`, Task Queue → `taskQueue.list()` by status, Brain → `brain.listDecisions()`, Rule Engine/Agent Manager/AI Provider Manager/Tool Manager/Workflow Engine/Scheduler → each one's own `list({status:"active"/"enabled"})`, Notification System → failed-delivery count), each just calling that module's own already-public read method — nothing here queries a database directly. A probe that throws is captured under `errors` in the `SystemStatus` response rather than aborting the rest, the same isolation principle as Event Bus subscribers. Counts are page sizes from a `limit`-bounded query, not `COUNT(*)` — documented as an approximation, not hidden. `GET /api/system/status` and read-only `GET /api/audit-log*` are both admin-only.
- **Settings** (`lib/settings`, `@workspace/settings`) — generic key-value settings at `system` scope (shared) or `user` scope (per user), stored in one `settings` table disambiguated by a database-unique `compositeKey` (`system:<key>` or `user:<userId>:<key>`) computed by the store itself. `set()` is a real `INSERT ... ON CONFLICT (composite_key) DO UPDATE`, not a read-then-write. `SettingsManager` enforces scope/userId pairing (`InvalidSettingScopeError` if mismatched) and nothing else — no per-key schema, no defaults; callers own what their keys mean. Wired in `artifacts/api-server/src/lib/settings-manager.ts`. `/api/settings/system*` is read-open/write-admin-only; `/api/settings/me*` is full read/write/delete for the caller's own settings only.
- **Agents** (10, referenced in Frontend Pages below) — not yet implemented as OS Core modules; currently frontend-only placeholders.
