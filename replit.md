## Where things live

- `lib/db` — Postgres schema (Drizzle), source of truth for all persisted state, including the `events` table (the Event Bus's durable store), the `tasks` table (the Task/Queue Manager's durable store), the `brain_decisions` table (the Brain's decision trail), the `rule_definitions` table (the Rule Engine's data-defined rules), and the `agents`/`agent_runs` tables (the Agent Manager's registry and run history).
- `lib/event-bus` — `@workspace/event-bus`: the OS Core Event Bus. The only inter-module communication channel in the system.
- `lib/task-queue` — `@workspace/task-queue`: OS Core's Task Manager + Queue Manager. Platform-independent (`TaskQueue` depends only on a `TaskStore` interface and a minimal local `EventPublisher` interface — no compile-time dependency on `@workspace/event-bus` itself). `DrizzleTaskStore` claims work via `SELECT ... FOR UPDATE SKIP LOCKED` so multiple worker processes can safely pull from the same queue.
- `lib/brain` — `@workspace/brain`: OS Core's Brain, the only decision maker. Platform-independent, same as `lib/task-queue` — `Brain` depends only on a `BrainStore` interface and minimal local `EventSubscriber`/`EventPublisher`/`TaskEnqueuer` interfaces, with no compile-time dependency on `@workspace/event-bus` or `@workspace/task-queue`. Rules (`DecisionRule`) are evaluated against every event they match; when more than one rule matches the same event, `@workspace/decision-engine` arbitrates which candidate(s) actually act. The Brain records every decision — `noop`, `superseded`, `actioned`, or `action_failed` — and, for the winning rule(s), enqueues a task or publishes a follow-up event.
- `lib/decision-engine` — `@workspace/decision-engine`: a small, pure arbitration library with no knowledge of events, tasks, or the Event Bus. Given a set of candidate actions competing for the same event, an `ArbitrationStrategy` picks which one(s) win. Ships `allMatchStrategy` (every candidate acts — the Brain's default, so adding arbitration never silently changes a deployment that only ever had one rule per event type), `firstMatchStrategy`, and `priorityStrategy`. `Brain` is its only consumer today; the Rule Engine reuses the same `DecisionRule.priority` field for its own compiled rules.
- `lib/rule-engine` — `@workspace/rule-engine`: data-defined rules on top of the Brain. A `RuleDefinition` (event pattern + JSON `Condition` tree + `ActionTemplate`, stored in the `rule_definitions` table) compiles into a `CompiledRule` and is hot-registered/unregistered against a running `Brain` as it's created, updated, enabled, disabled, or deleted through `/api/rules` — no restart required. Decoupled from `@workspace/brain` at compile time the same way `Brain` is decoupled from `@workspace/event-bus`: `RuleEvent`/`RuleOutcome` mirror `BrainEvent`/`RuleDecision`'s shapes exactly, and `RuleRegistrar` is the minimal registration capability `Brain` satisfies structurally.
- `lib/agent-manager` — `@workspace/agent-manager`: agent registration, lifecycle (`active`/`disabled`), and durable run history (`agents`/`agent_runs` tables). Deliberately does not know how to run an agent — `AgentExecutor` is the extension point the AI Provider Manager fulfills. `AgentManager.invoke()` fails loudly (`AgentExecutorNotConfiguredError`, surfaced as `501` from `POST /api/agents/:id/invoke`) rather than faking a response when no executor is wired up.
- `lib/ai-provider-manager` — `@workspace/ai-provider-manager`: provider configuration (`provider_configs` table — never stores a raw API key, only the name of the env var holding one) plus a uniform `ProviderClient.complete()` seam over whichever vendor a config points at. Ships a real client for Anthropic's Messages API (`AnthropicProviderClient`, injectable `fetch` for testability, genuine HTTP call in production). `ProviderBackedAgentExecutor` is the real `AgentExecutor` wired into `AgentManager` — turns an agent's `instructions` into the system prompt and the invocation input into a user message. Decoupled from `@workspace/agent-manager` at compile time the same way `RuleEngine` is decoupled from `Brain`.
- `lib/tool-manager` — `@workspace/tool-manager`: tool registration, schema-validated invocation, and an in-process handler registry (`tools` table stores the definition — name, description, a minimal JSON-Schema-like `inputSchema`, which `handlerName` runs it; the handler itself is code, registered via `registerHandler()`, never reconstructed from stored data). `invoke()` always validates input against `inputSchema` before running the handler, and best-effort broadcasts `tool.invoked`/`tool.failed` on the Event Bus (decoupled the same way `Brain`/`TaskQueue` are). Agent creation/update in `routes/agents.ts` validates every referenced `capabilities` entry against the Tool Manager's own registry, reusing `toolManager.list()` rather than duplicating any lookup.
- `lib/workflow-engine` — `@workspace/workflow-engine`: ordered, no-branching multi-step orchestration (`workflow_definitions`/`workflow_runs` tables). Each step (`tool`/`agent`/`task`/`event`) delegates to the real module that owns that capability through its own decoupled interfaces (`ToolInvoker`≈`ToolManager.invoke`, `AgentInvoker`≈`AgentManager.invoke`, `TaskEnqueuer`≈`TaskQueue.enqueue`, `EventPublisher`≈`EventBus.publish`) — it never reimplements any of them. Later steps reference earlier steps' outputs via whole-token templating (`renderTemplate`, a Workflow-Engine-specific implementation — its `{ input, steps }` context has nothing to do with Rule Engine's event-scoped renderer, so it isn't a fit for reuse). `run()` executes steps in order and stops at the first failure, recording every step's result either way.
- `lib/scheduler` — `@workspace/scheduler`: cron/interval-triggered workflow runs or task enqueues (`scheduled_jobs`/`scheduled_job_runs` tables). Ships a real, hand-rolled 5-field cron parser (`parseCronExpression`) and next-run-time calculator (`getNextRunTime`, brute-force minute-stepping — the same approach most cron libraries use internally, since cron is minute-granular by definition). `tick()` finds every active job whose `nextRunAt` has passed, dispatches through `WorkflowRunner`/`TaskEnqueuer` (decoupled, ≈ `WorkflowEngine.run`/`TaskQueue.enqueue`), records a `ScheduledJobRun` regardless of outcome, and reschedules `nextRunAt`. `start()`/`stop()` wrap `tick()` in a plain `unref()`'d interval timer for production; tests call `tick()` directly against an injectable clock. Single-process only — no distributed claim lock, documented as a real constraint.
- `lib/notification-system` — `@workspace/notification-system`: channel configuration and delivery tracking (`notification_channels`/`notification_deliveries` tables — the delivery record doubles as the in-app inbox, since "was this delivered" and "what's in the recipient's inbox" are the same fact for the one channel with nothing external to call). Ships `in_app` (a no-op beyond the delivery record itself) and `webhook` (a real HTTP POST, injectable `fetch` for testability). Optionally subscribes to the Event Bus for a `notification.requested` event type (decoupled `EventSubscriber` ≈ `EventBus.subscribe`, same pattern as `Brain`), so any other module can trigger a notification just by publishing an event. No email/SMS/chat-app channels ship built in — no credentials for any of those exist in this environment, so none are faked; `registerChannel()` is the real extension point once one is.
- `lib/audit-observability` — `@workspace/audit-observability`: two small, separate responsibilities. `AuditLogger` records to its own `audit_log` table — populated automatically by `middleware/audit.ts` (a global Express middleware recording every successful mutating request) rather than by editing every route file to call it. `ObservabilityService` aggregates a `SystemStatus` snapshot purely from independently-registered `MetricsProbe`s (one per module, registered in `artifacts/api-server/src/lib/audit-observability.ts`, each just calling that module's own already-public `list()`); a probe that throws is captured under `errors` without stopping the others.
- `lib/settings` — `@workspace/settings`: generic key-value settings at `system` scope (one value per key, shared) or `user` scope (one value per key per user). `set()` is a true `INSERT ... ON CONFLICT` upsert against a database-enforced unique `compositeKey` (`system:<key>` or `user:<userId>:<key>`), not a read-then-write. No schema for what a key/value should look like — callers own the meaning of their own keys.
- `lib/api-spec/openapi.yaml` — source of truth for the HTTP API contract. Run `pnpm --filter @workspace/api-spec run codegen` after editing it to regenerate `lib/api-zod` and `lib/api-client-react`.
- `artifacts/api-server` — Express entrypoint. Wires the Event Bus singleton (`src/lib/event-bus.ts`) and exposes it read-only via `GET /api/system/events`. Wires the Task Queue singleton (`src/lib/task-queue.ts`, built on the same Event Bus instance) and exposes it via `/api/tasks*`; also runs a periodic `reclaimStale()` sweep so tasks abandoned by crashed workers get requeued. Wires the Brain singleton (`src/lib/brain.ts`), registers its core decision rules at startup (`src/lib/brain-rules.ts`), and exposes its decision trail read-only via `/api/brain/decisions*`. Wires the Rule Engine singleton (`src/lib/rule-engine.ts`) on top of the same `brain`, syncing data-defined rules at startup and exposing full CRUD at `/api/rules*`. Wires the Agent Manager (`src/lib/agent-manager.ts`) with a real `AgentExecutor` — `ProviderBackedAgentExecutor` from the AI Provider Manager singleton (`src/lib/ai-provider-manager.ts`) — exposing agent CRUD/lifecycle/invoke/run-history at `/api/agents*` and provider config CRUD at `/api/provider-configs*`. Wires the Tool Manager (`src/lib/tool-manager.ts`, on the same Event Bus instance) exposing tool CRUD/lifecycle/invoke at `/api/tools*`; `routes/agents.ts` validates an agent's `capabilities` against it on create/update. Wires the Workflow Engine (`src/lib/workflow-engine.ts`) on top of `taskQueue`/`eventBus`/`agentManager`/`toolManager` simultaneously, exposing workflow CRUD/lifecycle/run/run-history at `/api/workflows*`. Wires the Scheduler (`src/lib/scheduler.ts`) on top of `workflowEngine`/`taskQueue`, started via `scheduler.start()` in `index.ts` after every other module and stopped during graceful shutdown; exposes job CRUD/lifecycle/run-history at `/api/scheduled-jobs*`. Wires the Notification Manager (`src/lib/notification-manager.ts`, on the same Event Bus instance, imported for its side-effect subscription in `index.ts`) exposing channel CRUD at `/api/notification-channels*` and send/inbox/mark-read at `/api/notifications*`. Applies the audit middleware globally in `app.ts` (before the router) and wires the Observability Service (`src/lib/audit-observability.ts`) with one probe per module above; exposes `GET /api/system/status` and read-only `GET /api/audit-log*`, both admin-only. Wires the Settings Manager (`src/lib/settings-manager.ts`) exposing `/api/settings/system*` (read open to any authenticated user, write/delete admin-only) and `/api/settings/me*` (every user's own, full read/write/delete). Applies `helmet` security headers globally and two `express-rate-limit` tiers (`middleware/security.ts`) — a general 300-req/15min limit on all of `/api`, plus a stricter 20-req/15min limit scoped to `/api/auth*` via a path-scoped `router.use("/auth", authRateLimiter)` in `routes/index.ts`, without modifying `routes/auth.ts` itself.
- `artifacts/octopus-os` — presentation-only web client. Business logic does not belong here.

## Architecture decisions

- **Brain-centric, event-driven core.** OS Core (Event Bus, Task/Queue Manager, and the Brain are all implemented) is the only decision maker. Agents and modules never call each other directly — every interaction is a published event on `@workspace/event-bus`, persisted to the `events` table before dispatch, so the event log is both the audit trail and the replay source.
- **Event Bus is in-process today, durable by design.** `EventBus` persists every event via `EventStore` (Postgres in production, in-memory for tests) before fanning it out to subscribers, and isolates handler failures per-subscriber so one bad handler can't break others or the publisher.
- **Task/Queue Manager is the durable work-handoff substrate built on the Event Bus.** `TaskQueue` (`lib/task-queue`) is how any module or agent hands off asynchronous work: `enqueue()` persists a task and publishes `task.created`; a worker `claim()`s it (atomic, exclusive, safe across processes), then `complete()`s or `fail()`s it. Failures retry with exponential backoff up to `maxAttempts`, then move to `failed`; every transition publishes an event (`task.started`/`completed`/`failed`/`retrying`/`cancelled`/`reclaimed`). This is intentionally decoupled from `@workspace/event-bus` at compile time via a local `EventPublisher` interface — `EventBus` satisfies it structurally — so the module stays testable and swappable in isolation.
- **The Brain reacts, it never polls.** `Brain` (`lib/brain`) evaluates every registered `DecisionRule` whose pattern matches an incoming event (exact type, `"namespace.*"`, or `"*"`) through a single Event Bus subscription — not one subscription per rule — so it always sees every rule interested in a given event before acting on any of them. For every matching rule it records a `BrainDecision` before doing anything else: `noop` if the rule declined, `action_failed` if the rule threw or its action failed, `superseded` if the rule wanted to act but the Decision Engine picked a different rule's candidate for the same event, or `actioned` once the winning rule's action (enqueue a task via `TaskQueue`, or publish a follow-up event via `EventBus`) succeeds. The Brain never re-evaluates rules against events it authored itself (source `"brain:*"` or `"os-core:brain"`), which is what keeps `publish_event` rules and the `brain.decision.made` broadcast from looping. Decoupled from `@workspace/event-bus`/`@workspace/task-queue` at compile time the same way `TaskQueue` is decoupled from `@workspace/event-bus`.
- **Arbitration is a separate, pure concern from decision-making.** `DecisionEngine` (`lib/decision-engine`) knows nothing about events, rules, tasks, or the Brain — it takes a list of `{ ruleName, action, priority, reason }` candidates and returns the subset that should act, per whichever `ArbitrationStrategy` it's configured with. The Brain is constructed with one (`allMatchStrategy` by default) and can be reconfigured at runtime via `DecisionEngine.setStrategy()`. Keeping this out of `Brain` itself means the same strategies are reusable wherever else "more than one thing wants to happen, pick the winner(s)" comes up.
- **Rules can be data, not just code.** `RuleEngine` (`lib/rule-engine`) lets a `RuleDefinition` — an event pattern, a JSON `Condition` tree (`eq`/`neq`/`gt`/`gte`/`lt`/`lte`/`contains`/`exists`, composed with `and`/`or`/`not`), and an `ActionTemplate` — be created through `/api/rules` and compiled into a `CompiledRule` the Brain registers, with no code deploy and no restart. `RuleEngine.update()`/`.enable()`/`.disable()`/`.delete()` always unregister the old compiled rule from the Brain before registering the new one (or not re-registering at all), since `Brain` only exposes register/unregister, never in-place mutation. Action payloads support whole-token templating only (a string that is *exactly* `"{{dot.path}}"` gets replaced with that field from the triggering event) — deliberately not a general expression language, so a rule definition can never be used to construct or evaluate arbitrary code.
- **Registration and execution are different concerns, on purpose.** `AgentManager` (`lib/agent-manager`) owns what an agent *is* (name, instructions, capabilities, active/disabled) and the durable history of every time one ran (`AgentRun`, always recorded once execution starts — `completed` or `failed`, never silently dropped). It has no idea how to actually run a model. `AgentExecutor` is the seam: `invoke()` throws a specific, catchable `AgentExecutorNotConfiguredError` when none is wired up, which `routes/agents.ts` turns into a `501` — an honest "not built yet" rather than a fake success.
- **One executor, no parallel paths.** `AiProviderManager` (`lib/ai-provider-manager`) supplies the real `AgentExecutor`: `ProviderBackedAgentExecutor` is constructed once in `artifacts/api-server/src/lib/agent-manager.ts` and passed straight into `AgentManager`'s constructor — the same interface the module's own tests satisfy with a fake. There is exactly one way an agent invocation reaches a model. A provider config's `apiKeyEnvVar` is the *name* of an environment variable, never the credential itself, so provider configs are safe to read back over `/api/provider-configs*`. Ships a real, HTTP-calling `ProviderClient` for Anthropic (`AnthropicProviderClient`) with an injectable `fetch` purely for unit-testing request construction — the production path performs a genuine call. Other vendors register via `AiProviderManager.registerFactory()` without touching existing configs or callers.
- **Tool definitions are data; tool handlers are always code.** `ToolManager` (`lib/tool-manager`) stores what a tool is called, its description, its `inputSchema`, and which `handlerName` runs it — never the handler's logic itself. `invoke()` validates input against `inputSchema` (a deliberately small JSON-Schema-like subset — primitives, enums, required object properties, array item types; no `$ref`/`oneOf`/custom formats, so a stored schema can only describe shape, never behavior) before a handler ever runs, and throws distinct, catchable errors (`ToolNotFoundError`, `ToolDisabledError`, `ToolInputValidationError`, `UnknownToolHandlerError`) for every way an invocation can fail before execution — none of those produce a `tool.invoked`/`tool.failed` broadcast, since nothing was attempted. Concrete handlers are registered in-process by whatever integrates that specific tool; none ship built in. Decoupled from `@workspace/event-bus` the same way `Brain`/`TaskQueue` are.
- **Workflows orchestrate; they never reimplement.** `WorkflowEngine` (`lib/workflow-engine`) runs a fixed, ordered sequence of steps — no branching, no loops, no parallelism — where every step type is a thin delegation to a real module through a decoupled interface: `tool` steps call `ToolInvoker` (≈ `ToolManager.invoke`), `agent` steps call `AgentInvoker` (≈ `AgentManager.invoke`), `task` steps call `TaskEnqueuer` (≈ `TaskQueue.enqueue`, fire-and-forget — a workflow doesn't wait for `task.completed`, by design; that would need mid-run Event Bus subscription, a real future enhancement), and `event` steps call `EventPublisher` (≈ `EventBus.publish`). A step whose dependency wasn't wired up throws `WorkflowDependencyMissingError` rather than silently skipping. Later steps reference earlier ones via whole-token templating against `{ input, steps: { <name>: { output } } }` — a separate implementation from Rule Engine's own renderer, since the context shapes have nothing in common; not a duplication of the *pattern* being wrong, just not the same data to render against.
- **The Scheduler is a trigger, not a runtime.** `Scheduler` (`lib/scheduler`) only ever does two things on a timer: run a workflow (`WorkflowRunner` ≈ `WorkflowEngine.run`) or enqueue a task (`TaskEnqueuer` ≈ `TaskQueue.enqueue`). `nextRunAt` is precomputed and stored on the job itself, so "what's due" is a plain `nextRunAt <= now` filter in `tick()`, not a live cron evaluation on every check. The cron parser (`parseCronExpression`/`getNextRunTime`) is a real, from-scratch 5-field implementation — wildcards, lists, ranges, step suffixes — not a wrapper around a library, since none was already a dependency here; it deliberately doesn't support `@daily`-style aliases or timezones, a documented gap rather than a silent one. `start()`/`stop()` are a thin `setInterval` wrapper so tests can call `tick()` directly against an injected clock instead of waiting on real timers. No distributed lock — fine for one process, a real constraint if ever run with multiple replicas.
- **A delivery record IS the inbox — no separate notifications table.** `NotificationManager` (`lib/notification-system`) records a `NotificationDelivery` for every send attempt, `sent` or `failed`, and that same record (`recipientUserId` + message + `read`) is what `GET /api/notifications` returns — there's nothing else an in-app inbox would need to store. `webhook` is the other built-in channel, a real HTTP POST with an injectable `fetch`; both are resolved through the same kind of factory registry `AiProviderManager` uses (`registerChannel()` for anything beyond the two built ones — no email/SMS/chat-app channel ships, since no credentials for any of those exist here, and faking one would be worse than not having it). Optionally subscribes to a `notification.requested` Event Bus pattern (decoupled, ≈ `EventBus.subscribe`) so any module can trigger a notification without depending on this package at all.
- **Audit trail via middleware, not per-route calls.** `middleware/audit.ts` records every successful (2xx/3xx) POST/PUT/PATCH/DELETE request to `AuditLogger` automatically, attached once and globally in `app.ts` before the router. The action name and resource type come from the matched route pattern (`req.route.path`), never the raw URL, so a UUID never leaks into an action name; the actor comes from `req.user`, read inside a `res.on("finish")` listener so it reflects whatever `requireAuth` populated earlier in that same request's chain. No existing route file needed to change for this — auditing thirteen-plus mutating resource types by editing thirteen-plus route files would have meant redesigning already-completed modules, which is exactly what this middleware approach avoids.
- **Observability aggregates; it never queries directly.** `ObservabilityService` (`lib/audit-observability`) has no idea what a `MetricsProbe` actually does — `artifacts/api-server/src/lib/audit-observability.ts` registers one per module (Event Bus, Task Queue, Brain, Rule Engine, Agent Manager, AI Provider Manager, Tool Manager, Workflow Engine, Scheduler, Notification System), each just calling that module's own already-public `list()`/`history()`. A probe's counts are page sizes from a `limit`-bounded query, not a true `COUNT(*)` — an honest approximation for a health snapshot, not an exact total past that page (see Gotchas). One probe throwing is captured under `errors` and never stops the rest from reporting, the same isolation principle the Event Bus uses for subscribers.
- **Settings are generic, uninterpreted key-value pairs.** `SettingsManager` (`lib/settings`) enforces only that `"user"` scope always carries a `userId` and `"system"` scope never does (`InvalidSettingScopeError` otherwise) — it has no idea what any particular key means, no schema per key, no defaults. Uniqueness is enforced at the database level on a computed `compositeKey` (`system:<key>` or `user:<userId>:<key>`), so `set()` is a genuine `INSERT ... ON CONFLICT DO UPDATE`, not an application-level read-then-write race.
- **Two "providers" concepts, deliberately kept apart.** `providersTable` (`lib/db/src/schema/providers.ts`, Postgres table `ai_providers`) is a pre-existing, per-user feature from before OS Core existed — it stores a raw `apiKey` column and backs the legacy `/api/providers*` routes and the mockup's "AI Providers" page. It is untouched. `providerConfigsTable` (`lib/db/src/schema/provider-configs.ts`, Postgres table `provider_configs`) is the new, OS-Core-integrated concept behind `/api/provider-configs*` — it never stores a raw key and is what actually powers agent execution. Don't confuse the two; if the legacy feature is ever migrated onto the real one, that's a deliberate future refactor, not something either module does implicitly.
- **API-first.** All HTTP contracts are defined in `lib/api-spec/openapi.yaml` first; `lib/api-zod`/`lib/api-client-react` are generated, not hand-written. The current exceptions are `system-events.ts`'s, `tasks.ts`'s, `brain.ts`'s, `rules.ts`'s, `agents.ts`'s, `provider-configs.ts`'s, `tools.ts`'s, `workflows.ts`'s, `scheduled-jobs.ts`'s, `notifications.ts`'s, `system-status.ts`'s, `audit-log.ts`'s, and `settings.ts`'s request validation, all hand-rolled until codegen is re-run in an environment with network access — see Gotchas.
- **UI is presentation only.** `artifacts/octopus-os` must never contain business logic; it consumes OS Core exclusively through the generated API client so the same Core serves web, mobile, desktop and future AI clients identically.

## Gotchas

- After editing `lib/api-spec/openapi.yaml`, run `pnpm --filter @workspace/api-spec run codegen` before relying on `@workspace/api-zod`/`@workspace/api-client-react` — they're generated output, not hand-written. As of the Task/Queue Manager module, the `/tasks*` paths are in the spec but codegen hasn't been re-run (no network access in the environment that built this), so `routes/tasks.ts` hand-rolls its own request validation, same as `routes/system-events.ts` already did. `routes/brain.ts`, `routes/rules.ts`, `routes/agents.ts`, `routes/provider-configs.ts`, `routes/tools.ts`, `routes/workflows.ts`, `routes/scheduled-jobs.ts`, `routes/notifications.ts`, `routes/system-status.ts`, `routes/audit-log.ts`, and `routes/settings.ts` add their paths the same way. Re-run codegen and switch all thirteen routes to the generated Zod schemas when network access is available.
- After adding a new file to `lib/db/src/schema`, run `npx tsc --build lib/db --force` before typechecking anything that imports `@workspace/db/schema` — `lib/db`'s composite TypeScript build cache (`tsconfig.tsbuildinfo`) doesn't always pick up a brand-new file in a downstream package's `--noEmit` check otherwise, and produces a confusing "no exported member" error that has nothing to do with the actual code.
- After adding/changing a `lib/db` schema file, run `pnpm --filter @workspace/db run push` against a real `DATABASE_URL` to apply it. This includes the `tasks`, `brain_decisions`, `rule_definitions`, `agents`, `agent_runs`, `provider_configs`, `tools`, `workflow_definitions`, `workflow_runs`, `scheduled_jobs`, `scheduled_job_runs`, `notification_channels`, `notification_deliveries`, `audit_log`, and `settings` tables — none have been pushed to any live database yet.
- `GET /api/system/status`'s counts are `.length` of a `limit`-bounded `list()` call per module (e.g. up to 500 tasks), not `SELECT COUNT(*)`. Fine as a health snapshot; once any module's real count exceeds its probe's limit, the number reported understates reality. A true count would need each store to grow a `count()` method — a real, scoped enhancement, not something to fake by raising the limit indefinitely.
- The audit middleware (`middleware/audit.ts`) only fires for POST/PUT/PATCH/DELETE with a 2xx/3xx response, attached globally before the router. It infers `resourceType`/`action` from the *matched* route pattern (`req.route.path`), so a request that 404s before matching any route (or that fails inside `requireAuth`, before the route handler runs) is never audited — nothing happened yet in either case, so there's nothing to record beyond what request-level logging already captures.
- No email, SMS, or chat-app-specific (Slack app, Teams, etc.) notification channel ships built in — no credentials for any of those exist in this environment. `webhook` covers Slack/Discord incoming-webhook-style integrations generically; anything needing an API key of its own needs a real `NotificationChannel` registered via `notificationManager.registerChannel()` once that credential exists, the same pattern as `AiProviderManager.registerFactory()`.
- The Scheduler's cron parser only supports standard 5-field expressions (minute hour day-of-month month day-of-week) with `*`, lists, ranges, and step suffixes — no `@daily`/`@hourly` aliases, no seconds field, no timezone qualifier (everything is evaluated in UTC). A cron expression using any of those will fail to parse with a clear error at job-creation time, not silently misbehave later.
- Running more than one api-server replica means more than one `Scheduler.start()` timer against the same `scheduled_jobs` table, with no distributed claim lock — a due job could fire once per replica in the same tick window. Fine today (single process); a real multi-replica deployment needs the same kind of claim mechanism `TaskQueue.claim()` already has.
- A `task` workflow step is fire-and-forget: `WorkflowEngine` records the enqueued task's id as that step's output and immediately moves to the next step — it does not wait for `task.completed`/`task.failed`. If a workflow needs a later step to depend on a task actually finishing, that's not supported yet; it would need the engine to subscribe to the Event Bus mid-run, which is a real gap, not an oversight to paper over.
- Creating a `provider_configs` row doesn't validate that its `apiKeyEnvVar` is actually set — that's only checked at call time, inside `AnthropicProviderClient.complete()` (and any future provider client), which throws a clear error naming the missing env var. Set the real credential as an environment variable with that exact name before invoking an agent that resolves to it.
- A `RuleDefinition`'s `ActionTemplate.payload` supports whole-token templating (`"{{payload.taskId}}"` → that field's value) but nothing more — no arithmetic, no string concatenation, no arbitrary expressions. This is a deliberate security boundary, not a missing feature: it's what keeps `/api/rules` (admin-only, but still a runtime-configurable surface that shapes what tasks get enqueued and what events get published) from becoming a way to execute arbitrary logic.
- `EventBus.publish()` never throws on a subscriber failure — failures are isolated per-handler and recorded on the event row (`handlerErrors`). If you need to know a side effect landed, don't await it via `publish`; subscribe to the relevant follow-up event instead.
- Any new OS Core module must go through `@workspace/event-bus` to talk to anything else — no direct imports between agent/module implementations.
- `TaskQueue.claim`/`complete`/`fail` in `routes/tasks.ts` currently authenticate workers the same way as any other API caller (`requireAuth`, any role) — there is no separate service/worker credential yet. Fine for now since nothing but the api-server itself and manual testing calls these; revisit before letting untrusted or third-party agents claim tasks.
- Any `DecisionRule` registered on the Brain must never subscribe to `"brain.decision.made"` or a pattern (`"*"`, `"brain.*"`) that could match it without being aware the Brain already filters out events it authored itself (source `"brain:*"`/`"os-core:brain"`) — that's what prevents `publish_event` rules and the decision broadcast from looping, not anything the rule itself needs to do.
- `pnpm run build` requires `PORT`, `BASE_PATH`, and `DATABASE_URL` to be set (Replit-style runtime config, checked at Vite/build-config load time for `mockup-sandbox`/`octopus-os` and at import time for `@workspace/db`) — set them (even to placeholder values for a typecheck-only pass) before running `pnpm run build` outside Replit.
- The periodic `reclaimStale()` sweep in `artifacts/api-server/src/index.ts` runs every 60s per process. If you ever run multiple api-server replicas, that's fine — `reclaimStale`'s requeue is a plain `UPDATE ... WHERE status = 'running' AND locked_at < ...`, safe to run concurrently — but it means stale-task detection latency is bounded by whichever replica's timer fires next, not exactly 60s.
- `Dockerfile`, `docker-compose.prod.yml`, and `.github/workflows/ci.yml` are real, complete configs, but none have been executed against real infrastructure from this environment — no Docker daemon, no GitHub Actions runner, and no reachable deployment target exist in the sandbox that produced them. They've been checked for syntax validity only (YAML parses, Dockerfile stages reference real, existing scripts). Treat "will it actually build/run/pass" as unverified until run for real — see `FINAL_DEPLOYMENT_GUIDE.md`.
- `apiRateLimiter`/`authRateLimiter` (`middleware/security.ts`) key by IP by default (`express-rate-limit`'s default `keyGenerator`). Behind a reverse proxy/load balancer, this only works correctly if `app.set("trust proxy", ...)` is configured to trust the right hop — not currently set, since the correct value depends on the specific proxy topology of wherever this is deployed.
